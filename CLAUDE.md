# Stargazer Barn ‚Äî Amazon Analytics Project

Stargazer Barn is an Amazon fresh-flower seller. This project contains dashboards and automation tools for tracking advertising performance, Share of Voice (SOV), and Best Seller Rank (BSR) on Amazon.

**To push any file live (e.g. `index.html`):**

```bash

# 1. Get current SHA (required by API to update existing file)
SHA=$(curl -s -H "Authorization: token $GITHUB_TOKEN" "$API_URL?ref=$GITHUB_BRANCH" \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('sha',''))")

# 2. Build payload (base64-encode the file)
python3 -c "
import base64, json
with open('/Volumes/GISCH SSD/CLAUDE/Stargazer/$GITHUB_FILE', 'rb') as f:
    content = base64.b64encode(f.read()).decode()
payload = {'message': 'Your commit message here', 'content': content, 'sha': '$SHA', 'branch': '$GITHUB_BRANCH'}
print(json.dumps(payload))
" > /tmp/github_payload.json

# 3. Push
curl -s -o /tmp/github_response.json -w "%{http_code}" \
    -X PUT "$API_URL" \
    -H "Authorization: token $GITHUB_TOKEN" \
    -H "Content-Type: application/json" \
    --data @/tmp/github_payload.json
```

HTTP 200 = updated, 201 = created. The automated BSR cron (`run_bsr.sh`) uses this same pattern to push `dashboard_data.js` every morning at 6:30am.

## Monthly Ads + Sales Data Ingest

Run this every month when the user provides new raw exports. Two files are needed:

1. **Ads report** ‚Äî Amazon Advertising Console ‚Üí Search Term report ‚Üí Monthly, all campaigns. Filename pattern: `Stargazer_-_ALL_REPORT_-_Monthly_NEW.csv`
2. **Sales report** ‚Äî Amazon Vendor Central ‚Üí Sales ‚Üí ASIN/Manufacturing/Retail view, 1-month custom range. Filename pattern: `Sales_ASIN_Manufacturing_Retail_UnitedStates_Custom_M-D-YYYY_M-D-YYYY.csv`

### What the ingest updates

All seven data arrays in `dashboard_data.js` plus the three intermediate CSVs:

| Target | How built |
|--------|-----------|
| `campaign_data` | Ads CSV grouped by Campaign name + Product Group |
| `sku_data` | Ads CSV grouped by SKU + Product Group |
| `search_term_data` | Ads CSV grouped by Search Term + SKU ‚Äî **top 50 unique terms per month by spend** |
| `time_series` | Total monthly rollup ‚Äî ad metrics from ads CSV, shipped metrics from sales CSV |
| `asin_performance` | SKU-level ad metrics merged with per-ASIN shipped revenue/COGS/units from sales CSV |
| `daily_series` | Daily ad aggregates (spend, sales, clicks, impressions, purchases) ‚Äî drives the Overview trend chart |
| `weekly_series` | Same fields rolled up by week (Monday-start) ‚Äî drives the Overview trend chart weekly view |
| `processed_ads_data.csv` | Raw ads rows appended with `Month_Period` and `Product Group` columns |
| `processed_sales_data.csv` | Raw sales rows appended with `Month`, `Month_Name`, `Product Group`, `Month_Key` columns |
| `monthly_performance.csv` | Product Group √ó Month rollup with TACoS, CAC, ROAS |

**`daily_series` / `weekly_series`**: built from the daily `Date` column in the raw ads CSV. Dates are parsed from Amazon's format (`Apr 01, 2026`) ‚Üí ISO (`2026-04-01`). Weeks use Monday as the start day.

### How to run the ingest (run inline Python ‚Äî no saved script needed)

1. Place both raw export files in `~/Downloads/` (or note their actual paths).
2. Open a Claude session in this project directory.
3. Tell Claude: *"Here are the two new files for [Month] data: [ads path] [sales path]. Please ingest as you always do to update all data."*
4. Claude will run the ingest inline, verify the summary numbers, update all arrays in `dashboard_data.js` and the three intermediate CSVs, then push to GitHub.

### Key ingest logic

- **Idempotent**: the ingest first strips any existing rows for the target `Month_Period` before appending, so re-running with the same month is safe.
- **search_term_data policy**: only the top 50 unique search terms per month (by total spend) are kept to control file size. Each term may produce multiple rows (one per SKU/Product Group), so 50 terms ‚Üí ~400‚Äì500 rows per month.
- **TACoS** = `Total cost / Shipped Revenue √ó 100`
- **CAC** = `Total cost / Purchases`
- **ROAS** = `Ad Sales / Total cost`
- **CTR** = derived from `Impressions √ó CTR%` in the raw file (Clicks column is reconstructed)
- **Organic-only ASINs**: some ASINs have shipped revenue/COGS in the sales file but zero ad spend. These must still be added to `asin_performance` (with all ad metrics zeroed out) so that the Business Performance tab shows correct COGS and Revenue totals. After building `asin_performance` from the ads data, iterate the sales CSV and add any ASIN not already present as a zero-spend row with real shipped data.
- **Annual Plan tab**: reads actuals from `time_series` dynamically for any `2026-XX` month present ‚Äî no code changes needed when adding new months. The tab will automatically show and color-code each new month as its data lands in `time_series`.
- **Overview trend chart**: uses `daily_series` (daily view), `weekly_series` (weekly view), and `time_series` (monthly view). All three must be updated each ingest or the chart will stop at the prior month. Build `daily_series` from the `Date` column in the raw ads file; roll up weekly from those daily rows.

## Important Notes

- Do not modify the 22-ASIN list without also updating `sov_updater.py` and the ASIN_GROUP mapping above
- `dashboard_data.js` is generated ‚Äî do not hand-edit it; always use the ingest process or updater scripts
- BSR and SOV data accumulate daily; rerunning on the same date overwrites that day's entry
- Raw CSV/XLSX files are source data; never delete them
- The `search_term_data` array is intentionally capped at top 50 terms per month by spend ‚Äî this keeps `dashboard_data.js` under 1 MB. The backup `.bak` file always holds the previous state.
